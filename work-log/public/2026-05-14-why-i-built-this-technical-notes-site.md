## One-line takeaway

**EN.** I built this site to turn my learning, research notes, and engineering practice into a durable public record.

**KO.** 이 사이트는 내가 공부한 내용, 연구 노트, 엔지니어링 실천을 장기적으로 남길 수 있는 공개 기록으로 만들기 위해 시작했다.

## Why this site exists

**EN.**<br>
I do not want this site to be a casual diary. I want it to be a technical record that shows how I learn, how I organize ideas, and how I connect research-level concepts with practical engineering constraints.

My main interests are robotics AI, AI systems performance, GPU optimization, and research-driven engineering. These areas are tightly connected: a robotics AI model is only useful in the real world when it can run reliably, efficiently, and within system-level constraints.

This site is my public layer for that process.

**KO.**<br>
나는 이 사이트를 단순한 일기장으로 만들고 싶지 않다. 이곳은 내가 어떻게 배우고, 어떻게 생각을 정리하고, 연구 수준의 개념을 실제 엔지니어링 제약과 어떻게 연결하는지를 보여주는 기술 기록이 되어야 한다.

내 주요 관심사는 robotics AI, AI systems performance, GPU optimization, 그리고 research-driven engineering이다. 이 분야들은 서로 강하게 연결되어 있다. 로봇 AI 모델은 실제 시스템 안에서 안정적으로, 효율적으로, 그리고 주어진 제약 안에서 동작할 때 비로소 의미가 있다.

이 사이트는 그 과정을 공개적으로 정리하기 위한 공간이다.

## What I plan to write

**EN.**<br>
I plan to use this site for four main types of notes.

| Section | Purpose |
| --- | --- |
| Work Log | Public-safe reflections on learning and engineering practice |
| Study | Technical notes on systems, GPU optimization, robotics, and math |
| Paper | Research paper summaries and implementation-oriented reading notes |
| Project | Reproduction logs, toy implementations, benchmarks, and tools |

The goal is not to write perfect articles from the beginning. The goal is to accumulate useful, searchable, and technically honest notes over time.

**KO.**<br>
이 사이트에는 크게 네 종류의 글을 쓸 예정이다.

| Section | 목적 |
| --- | --- |
| Work Log | 공개 가능한 학습 및 엔지니어링 회고 |
| Study | 시스템, GPU 최적화, 로보틱스, 수학 관련 기술 노트 |
| Paper | 논문 요약과 구현 관점의 읽기 노트 |
| Project | 재현 실험, toy implementation, benchmark, 개인 도구 기록 |

처음부터 완벽한 글을 쓰는 것이 목표는 아니다. 시간이 지날수록 유용하고, 검색 가능하며, 기술적으로 정직한 기록을 쌓는 것이 목표다.

## How I want to write

**EN.**<br>
I want each note to be written with a top-down structure:

1. What problem am I trying to understand?
2. Why does it matter in a real system?
3. What is the key technical idea?
4. What should I check experimentally or in code?
5. What did I learn?

This structure is especially important for AI systems performance work. It is easy to jump directly into kernels, counters, tools, or implementation details. But without a clear system-level question, low-level optimization can become unfocused.

**KO.**<br>
나는 각 글을 가능한 한 top-down 구조로 쓰고 싶다.

1. 내가 이해하려는 문제는 무엇인가?
2. 이 문제가 실제 시스템에서 왜 중요한가?
3. 핵심 기술 아이디어는 무엇인가?
4. 실험이나 코드에서 무엇을 확인해야 하는가?
5. 내가 배운 점은 무엇인가?

이 구조는 특히 AI systems performance 작업에서 중요하다. 커널, counter, tool, 구현 세부사항으로 바로 들어가는 것은 쉽다. 하지만 명확한 시스템 수준의 질문이 없으면 low-level optimization은 방향을 잃기 쉽다.

## Public-safe boundary

**EN.**<br>
This is a public website. Therefore, I will keep a strict boundary between public technical notes and private work logs.

I may write about:

- Public papers
- Public documentation
- Open-source code
- Personal toy examples
- General engineering lessons
- Reproduction experiments using public resources

I will not write about:

- Internal code
- Non-public benchmarks
- Private system architecture
- Internal logs, traces, screenshots, or profiler outputs
- Confidential project names
- Details that should belong only to a private work log

This boundary is not a formality. It is part of responsible engineering practice.

**KO.**<br>
이 사이트는 공개 웹사이트다. 따라서 공개 기술 노트와 비공개 work log 사이의 경계를 엄격하게 유지할 것이다.

쓸 수 있는 내용은 다음과 같다.

- 공개 논문
- 공개 문서
- 오픈소스 코드
- 개인 toy example
- 일반화된 엔지니어링 lesson
- 공개 자원 기반 reproduction experiment

쓰지 않을 내용은 다음과 같다.

- 내부 코드
- 비공개 benchmark
- 내부 시스템 구조
- 내부 log, trace, screenshot, profiler output
- 비공개 프로젝트명
- private work log에만 남겨야 하는 세부사항

이 경계는 형식적인 규칙이 아니다. 책임 있는 엔지니어링 실천의 일부다.

## Long-term direction

**EN.**<br>
Over time, I want this site to become a public index of my growth toward robotics AI systems performance engineering. I am interested in the full path from model-level ideas to deployment-level constraints: perception-action models, robot foundation models, diffusion or flow-based policies, GPU inference optimization, profiling, latency analysis, and hardware-aware acceleration.

The long-term value of this site will come from consistency. A single note is small, but a well-maintained collection of notes can reveal how someone thinks.

**KO.**<br>
장기적으로 이 사이트가 robotics AI systems performance engineering을 향한 나의 성장 과정을 보여주는 공개 index가 되기를 바란다. 나는 model-level idea에서 deployment-level constraint까지 이어지는 전체 경로에 관심이 있다. 예를 들어 perception-action model, robot foundation model, diffusion 또는 flow 기반 policy, GPU inference optimization, profiling, latency analysis, hardware-aware acceleration 같은 주제들이다.

이 사이트의 장기적인 가치는 꾸준함에서 나올 것이다. 하나의 글은 작지만, 잘 관리된 글의 모음은 한 사람이 어떻게 생각하는지를 보여줄 수 있다.

## Closing note

**EN.**<br>
This first post is intentionally simple. It defines the purpose of the site and confirms that both English and Korean can be used naturally in the same technical note.

**KO.**<br>
이 첫 글은 의도적으로 단순하게 썼다. 이 사이트의 목적을 정리하고, 하나의 기술 노트 안에서 영어와 한국어를 자연스럽게 함께 사용할 수 있는지 확인하기 위한 글이다.
